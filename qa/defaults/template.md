# Atlas 笔记模板(default)

给 LLM 的生成指示。目标读者：**第一次读这个代码库的工程师**。中文，人话优先；行文用全角标点（，。；：），代码、URL、`code:` 锚点等技术片段除外。

本模板**自包含**：全部规则连同内嵌骨架可整份拷贝到任何仓库使用。仓库可自带 `.atlas/templates/default.md` 整体替换本出厂件；替换版建议在此位置列出**本仓样板**（本仓库里几篇公认写得好的笔记路径——逐段导读金标准、目录页/文件页分工示范、概念枢纽页示范各一），写作前先读样板对齐手感。

## 语气

参考手册的冷静陈述。「人话」指用平实语言解释概念，不是说书或导览——事实直接说，重要性靠事实本身和结构表达，不靠修辞渲染。具体禁令：

- 禁止设问—揭晓句式：「答案是——」「有意思的是」「值得注意的是」这类悬念铺垫，一律删掉直接陈述。
- 禁止说书人自指与导游口吻：「挑几个说」「我们来看」「值得在这里看的是」。
- 禁止拟人和情绪化修辞：函数没有「脾气」、代码不「敢」做什么、不搞「宁可…也不…」排比。
- 禁止对读者下指令：「要单独记住」「一定要注意」。
- 措辞选日常语：「不传/不配置」而不是「缺省」；「可选参数不传会怎样」必须先讲清「这个参数是干什么用的」再讲不传的后果——只写省略效果不写用途的选项清单不合格。
- 标题是内容的直白陈述(名词短语或短句)：禁止「先读/先看/先说」式引导标题；冒号副标题只许补充事实(「validateConfig：启动期体检」)，禁止吊悬念(「X：要做哪些决定」)。

## 结构

1. **概览** — 两三段完整段落：这个文件/目录是什么、为什么存在、在整体里扮演什么角色。**第一段先回答「谁在用这个东西」**——调用方、客人、触发场景，用读者已有的经验锚定（「浏览器里的用户点的每个按钮」「拨回来的 runner」），再引出机制；默认读者不认识本仓库的任何角色名，每个机制名词（gateway、组合根、outbox…）出场前必须先被这样立起，或当句用一句人话立起。所有术语要么用 `.atlas/glossary.md` 里的规范名(一物一名，禁止同义词)，要么就地解释。概览末尾可放一张全局鸟瞰 mermaid(线性流程用 `flowchart LR`)——但仅当真有全局结构可画；把小节标题串成一条箭头链的图没有概念增益，不要画。
2. `---` 分隔线。
3. **interface 小节**(当文件导出较多 interface / type 时必写) — 在逐段导读之前开一节「interface 总览」。这一节必须**自包含**：读者只靠笔记就能建立契约地图。三个组成部分：

   **(a) 全景先行，禁止数数腔**：节的开头用一句业务话立起全貌（「必填项决定这台 daemon 对外暴露哪些面，可选项每个是一条部署决定」），**禁止**「导出 N 个顶层形状」「1 个接线对象 + 1 个解析结果形状」式的清点开场——数量本身是零信息。嵌套关系用平坦分组清单优先；只有嵌套真的深且交错时才画树图，把类型定义原样画成方框连线没有概念增益。

   **(b) 每簇讲业务决定，不许翻译标识符**：按逻辑簇拆 `##### [簇名](code:起始..结束)`，簇下写「这个形状表达什么部署/产品决定」——每个枚举值是什么姿态、什么时候选它、null 关掉的是哪块功能、谁消费。**「`SignupMode` 管注册门」这种把类型名翻译成中文的句子 = 零信息，不合格**；合格线是每句话说出标识符本身说不出的东西（删掉这句读者理解不变的，删）。语义必须从当前源码及其注释核实，源码撑不住的断言（如「云上选哪档」）不写。

   **(c) 转译嵌入，绝不粘贴源码**：`![说明文字](code:起始..结束)` 会在渲染时从当前源码实时切片高亮（单标记自动取整个括号配平块，如 `![一条问题 + 级别](code:ConfigIssue)`）。**禁止在围栏代码块里手工粘贴仓库源码**——那是会腐烂的副本。嵌入的使用纪律：只用在「代码形状本身就是论点」的小类型（判别 union 的样板、几行的 interface）；大簇保持 `[标题](code:...)` 链接锚点，全嵌等于把笔记退化回文件本身。正文反引号类型名仍会自动变成跳转锚点。
4. **概念小节**(可选) — 贯穿多段代码的横切概念(如统一的解析纪律、三值语义)单独成节讲透，**配图放在这里**：图服务于概念(决策树、状态、拓扑)，不是装饰。讲清楚之后，逐段导读里只需引用，不必重复。
5. **逐段导读** — 按代码从上到下的顺序，每个逻辑块一个小节，保证读者能从头到尾把文件走一遍：

   ```
   #### [块的名字](code:起始标记..结束标记)

   两三句完整段落：是什么、为什么存在、什么时候需要读它/判断标准。
   ```

   `code:` 是内容锚点：标记为符号名或行内文字，**不能含空格**(也不要带 `export const` 这类前缀，直接用符号名)；范围 = 起始标记行到结束标记行的前一行；单标记 = 单行；定义行优先解析。正文反引号提到本文件的符号会自动变成跳转锚点，无需手写。

   密度纪律：每小节两三句完整段落，一句只装一个事实；概念已在前面的小节立住的，这里只引用不重讲。读者应该能只靠逐段导读把文件从头走到尾，不需要来回跳。
6. `---` 分隔线。
7. **进阶细节 callout**(可选，没有高密度事实就不要硬凑)：

   ```
   <div class="callout">

   💡 **进阶细节** — 一句话列出覆盖的主题。

   **加粗引导句。** 完整句子的段落……

   </div>
   ```

8. **HTML 解释块**(鼓励) — 笔记里可以写任意 HTML + inline style，viewer 原样渲染。当一个概念「画出来比说出来清楚」而 mermaid 的死板布局又表达不了时，**鼓励**直接用 HTML 做生动的解释性排版：字节/内存布局图、带标注的时间线、着色分区的对照矩阵、嵌套关系的方框图……判断标准仍是概念增益：布局要承载语义，不是装饰。三个现成 class 免写样式可直接用：`.callout`(旁注/进阶)、`<details><summary>`(长内容折叠)、`.cols`(双栏对照)；此外的自定义排版用 inline style 自包含(viewer 是浅色主题，配色对齐 `#fbfbfa` 底/`#e7e5e1` 边框/`#3d6b54` 强调即可)。机械规则：块级标签与内部 markdown 之间必须空行，否则 markdown 不解析。

## 概念枢纽目录页(2026-07-05 capabilities 页返工总结)

判据：一个目录页若要立起 **3 个以上本仓术语**，它就是「概念枢纽」，用下面的配方而不是普通目录页结构。

1. **矛盾句起手**（同全局规则），一段就够，不在这里立名词。
2. **一次真实调用的走查**代替名词课：从当前源码里挑**一条真实存在、且实现足够短的调用**（先核实那几行代码真的在），做成逐站卡片（HTML）。每站两层字：**黑字写具体发生的事**（真实路径、真实代码行、真实标识符），**辅色小字才给术语定义**——读者先看见事实、再收到名词，永远不反过来。站数 ≤ 6。「薄/简单」这类形容词，用贴出真实的三行代码代替。
3. **两个反例**（各一张暖色卡片）：① 不走这套机制的最顺手捷径长什么样 + 代价逐条到账（每条都要具体：什么控制不到它/谁看不见它/得自己抄什么）；② 边界外的诱惑——**为什么会有人这么做**（缺了什么、赶什么）+ 绕过去具体坏在哪 + 正确动作是什么。反例必须明确标成假设（「假如/诱惑是」），绝不能写得像已存在的代码——真实性纪律优先。
4. **制度回报段**收尾（这样做换来了什么，一段），可配一张对照式视觉块。
5. 之后才是常规的文件清单。

内嵌骨架（自包含，任何仓库可直接套；`〈〉`是待填槽位，示例站取自一个 RPC 柜台架构，替换成你仓库的真实调用即可）：

```html
〈矛盾句一段：这个目录解决什么问题，没有它哪里会坏〉

跟着一次真实点击走一遍，路上的名词顺手就认识了。〈触发场景〉：

<div style="border:1px solid #e7e5e1; background:#fbfbfa; border-radius:6px; padding:10px 14px;">
<b>① 〈站名：谁做了什么〉</b>　<code>〈真实请求/调用行〉</code><br>
<span style="color:#3d6b54; font-size:.88em;">〈这一站顺手立起的术语 + 一句定义〉</span>
</div>
<div style="text-align:center; opacity:.4;">↓　（〈站间备注，可省〉）</div>
<div style="border:1px solid #e7e5e1; background:#fbfbfa; border-radius:6px; padding:10px 14px;">
<b>② 〈查表/分发站〉</b>　<code>〈"名字" → 登记项〉</code>…
</div>
〈…至多 6 站，最后一站落在真正干活的部门〉

<div style="border:1px solid #d8b4ae; background:#fdf8f7; border-radius:6px; padding:10px 14px;">
<b>反例一：〈最顺手的捷径〉。</b>〈为什么诱人〉。代价逐条到账：〈具体后果一〉；〈二〉；〈三〉。
</div>
<div style="border:1px solid #d8b4ae; background:#fdf8f7; border-radius:6px; padding:10px 14px;">
<b>反例二：〈边界外的诱惑〉。</b>为什么会发生：〈缺了什么〉。绕过了什么：〈具体保护〉。正确动作：〈给机制扩容而不是绕过〉。
</div>

〈制度回报段：这套机制换来了什么，一段〉
```

防冗余（grok 批量时逐条执行）：走查只出现在枢纽页，子文件页用一句话衔接、**绝不重复走查或重立术语**；枢纽页立过的术语进 glossary，子页直接使用；走查卡片本身就是流程图，**不要再画一张同内容的 mermaid**；每个视觉块单独过概念增益门槛。

## 分层与海拔(2026-07-05 http 两页返工总结)

- **概览必须先陈述矛盾，调用方清单不算**：第一段要能回答「这个文件在解决什么问题/没有它哪里会坏」，且这句要写在最前（「这个文件解决一个问题：机器不会登录」）。只罗列「谁打哪个路径/谁调用哪个函数」是清单不是问题陈述——调用方视角是手段，矛盾陈述才是目的，两者都要有。协议细节、多阶段机制、字段语义一律不进概览，降到自己的小节。

- **目录页与文件页分工，不许同海拔重复**：目录页是地图——谁来敲门、有哪几块、块与块的关系、一张全景图、贡献纪律；文件页是该文件自己的故事，开头一句话衔接目录页（「目录页介绍了五扇门；这个文件是把它们钉成一条链的组装点」），**不重复目录页已画的图和已讲的清单**。连续读两页不应看到同一条流程的两种画法。
- **正文海拔纪律**：读者读完正文（不含 callout）应能复述「这东西为什么存在 + 两三个关键决定」；字段级契约语义、跨租户反查路径、边角实现细节一律下沉进阶 callout。callout 不是垃圾桶——正文讲决定，callout 讲机制细节，两边都要成文。
- **不变量必须带后果**：顺序/约束类知识写成「规则 + 违反了会怎样」（「SPA 静态必须在 transport 之前——否则客户端路由上刷新页面拿到的是 JSON 404」）。只有断言没有后果的规则，读者既记不住也无法验证。
- **图不限数量，但每张单独过门槛**：判断标准始终是概念增益——这张图讲的结构文字说不清吗？同一篇里多张图各讲各的概念是好的；不合格的是同一条流程换个画法再画一遍（本页与目录页之间同理），以及把类型定义、小节目录这类清单式内容画成框图。

## 禁令

- **禁止讲 atlas/viewer 机制本身**：「标题可点击」「跳到右侧预览并高亮」「repo-atlas check 会报 rot」这类 viewer 说明书话术一句都不许进正文——锚点和嵌入是隐形基础设施，说明文字要写成链接不渲染时整句读起来也通顺的普通文句。笔记只讲被描述的代码。
- **禁止零信息句**：把标识符翻译成中文再念一遍(「`X` 管 Y」「`Z` 负责 W」而 Y/W 就是名字的直译)不是内容。检验：删掉这句，读者对代码的理解少了什么？少了零就删。
- 禁止行号(如 L114)——会腐烂，定位一律用 `code:` 锚点。
- 禁止在围栏代码块里粘贴仓库源码——用 `![说明](code:起始..结束)` 转译嵌入(见 interface 小节)。围栏只用于笔记自有的示意（mermaid、env 示例等源码里不存在的内容）。
- 禁止受众标签(「给回头查资料的人」「初读可跳过」这类免责说明)——用结构和视觉分区解决。
- 列表纪律是双向的：叙述（有因果、递进、转折的内容）用段落，禁止拆成 `-` 列表；反过来，**平行短枚举必须用列表**——N 个「字段名 — 含义」「选项 — 用途 — 不传后果」这类结构平行、各自独立成立的条目，一项一行，禁止塞进一句话的顿号串（读者要逐项查，扫不了顿号串）。判据：条目之间没有逻辑连接词、调换顺序不影响理解 → 列表；有「因为/所以/但」→ 段落。
- 禁止没有概念增益的配图；位置纪律：概览放全局鸟瞰，小节放局部概念图。
- 不硬凑三段式：小文件可以只有概览(+少量小节)，没有值得写的 callout 就不写。

## 文件类型微调

- **测试文件**：不写笔记——`**/*.test.*`、`**/*.spec.*`、`__tests__` 已被 `config.json` 排除出 atlas；测试锁定的行为契约写进**被测模块**自己的笔记。正文提及测试时用白话（「MCP 集成测试」），不要写成会生成路径链接的反引号文件名。`test-helpers/` 不是测试，是搭建微缩系统的真代码，照常写。
- **目录笔记**(`__dir__.md`)：概览 + 每个子项一句话定位(子项名用反引号，自动成链接)；frontmatter 的 `order` 字段维护阅读顺序，不要动它。
- **配置/清单类**：只写非显然点(为什么排除了 X、为什么这个依赖是 devDependency)，不复述字段。**注册表/花名册/barrel 汇出这类"结构即全部内容"的代码文件同属此类**——笔记三段封顶：它收什么交给谁、一两条带后果的维护纪律、完。给简单文件写长笔记本身就是失败；枢纽页已讲过的概念在子页一律不重讲。

## 调查要求(反片面)

写 X 的笔记之前，必须：

1. 通读 X 的当前源码(不是靠旧笔记的印象)；
2. 读 X 所在目录的 `__dir__.md` 和相邻文件名，搞清 X 在局部的位置；
3. grep 找出**谁 import X、X import 谁**，抽读 2–3 个真实调用点，确认「它实际被怎么用」；
4. 旧笔记的说法与代码冲突时，以代码为准；
5. 发现 glossary 缺术语时不要私自加词条，在最终 summary 里列出建议新增的术语；
6. **可疑发现必须写进笔记，不许吞掉**：调查中发现「零调用者（死代码）」「代码注释宣称的行为与实际接线不符（意图≠现实）」「签名/校验只有生产侧没有消费侧」这类存疑事实时，在笔记正文加一段 `⚠ **现状存疑**`（放概览之后），如实写"证据是什么、缺口是什么"，并在最终 summary 里单独上报。**绝不把注释里的意图当成已发生的现实写进正文**——描述必须以"谁真的调用了它"为准。

## 机械规则

- 笔记文件在 `.atlas/notes/<仓库路径>.md`(目录是 `.../<dir>/__dir__.md`)。
- 只重写 frontmatter(`--- ... ---`)之后的正文；frontmatter 的 hash/anchor/stamped/order 一律不动。
- 不创建新笔记文件，只重写已存在的。
