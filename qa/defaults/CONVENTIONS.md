# .atlas 笔记规范

写笔记的 agent 必读。工具是 [repo-atlas](https://github.com/KUD-00/repo-atlas)：
`status --json` 拿 missing/outdated 清单 → 读代码写笔记 body → `stamp <具体路径>` → 页面自动刷新。

## 硬规则

1. **语言**：中文行文；标识符、术语、包名保留英文原样。
2. **frontmatter**（`---` 之间的 hash/stamped）由 `stamp` 命令管理——不要手写、不要改动；新笔记可以完全不写 frontmatter。
3. **只 stamp 自己刚审过的路径**（`repo-atlas stamp packages/x packages/x/src/y.ts`），永远不要裸跑 `stamp`/`stamp --all`——那会把没审过的笔记也盖成"新鲜"。
4. **outdated 的路径做增量修订**：先 `git log --oneline --since=<stamped时间> -- <路径>` 看改了什么，只修正受影响的句子，不要整篇重写。
5. 笔记写"代码里读不出来的东西"：为什么存在、不变量、坑、谁依赖它。**不要**复述代码逐行做什么、不要写文件清单（树就在左边）。
6. **笔记必须自包含**：不许引用或链接任何 spec / plan / 设计文档路径。spec 会移动、会过时，读者不应需要二跳——把 spec 里的**结论**用一两句写进笔记本身。可以引用的只有：代码路径、包名、本 atlas 里的其他目录/文件（它们会自动成链）。
7. **术语从 `.atlas/glossary.md` 取用**：写"L0"、"outbox"、"lease" 这类专业词之前先查词汇表，按表里的写法用（页面会自动标注并悬停出定义）。要用表里没有的新术语，先在词汇表加条目再用——防止各笔记各说各话。
8. **依赖/被依赖不用写**：页面会从 import 自动分析出 "imports → / ← imported by" 并成链。笔记里只写依赖关系的**为什么**（如"X 依赖 Y 是为了 Z 不变量"），不要罗列依赖清单。

## 目录笔记模板（`__dir__.md`）

```markdown
一到三句：这个目录是什么、为什么单独存在、在哪条分层轴上（L0/L1/…、plane）。

## 分工
- `子项A` — 承担什么角色（只列承重的，不求全）
- `子项B` — …

## 关系与不变量
- 被谁依赖 / 依赖谁（跨包边界写清楚）
- 必须成立的约束（如"payload 对队列不透明"）

（可选）mermaid 图，见下方图规则。
```

## 文件笔记模板（`<name>.md`）

```markdown
一到三句：这个文件是什么、为什么存在。

- 不变量/约束：…
- 被谁调用：…
- 坑：…（没有就整条省略）
```

- 纯 re-export 的 `index.ts`、生成物目录（`drizzle/`、`dist/`）：一句话即可。
- 测试文件：写"测的是哪些不变量"，不是"有哪些 case"。

## 图规则（mermaid）

笔记里的 ` ```mermaid ` 围栏会在页面上渲染成图。**默认不画**；只有满足以下之一才画：

| 情形 | 用什么图 |
|---|---|
| 状态机 / 生命周期，≥4 个状态 | `stateDiagram-v2` |
| 跨组件时序，≥3 个参与方 | `sequenceDiagram` |
| 依赖/数据流结构非显然（有环、有扇出） | `flowchart LR` |

- 图不限数量，每张单独过概念增益门槛（同一条流程不画两遍，含目录页与文件页之间）；单图节点数 ≤ 12；节点标签用代码里的真实名字。
- 线性的三步流程、目录清单、纯装饰——不要画，文字更便宜也更准。
- 图必须能通过 mermaid 解析（渲染失败页面会红框报错）；不确定语法就不画。
