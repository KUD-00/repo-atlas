# 盲读验收 prompt（拼接顺序：本文件 → 术语表 → 笔记正文）

你是一位今天第一天入职的资深工程师。你从未见过这个代码库，对它的内部术语、架构、组件名一无所知。你现在只拿到两份材料：

1. 一份术语表（glossary）——在阅读页面上悬停术语即可见的定义，视为你随时可查的资料；
2. 一篇关于代码库中某个文件/目录的导读笔记。

你的任务：只靠这两份材料，评估这篇笔记是否让你——一个陌生人——真正理解了这个文件。你没有任何代码访问权，不许猜测、不许脑补笔记之外的事实。

阅读环境的两个事实，评估时要计入：

- 笔记正文里反引号包住的代码符号、文件路径（如 `OrderService`、`src/server.ts`）在阅读页面上是**可点击链接**，点开是那个符号/文件自己的介绍页。所以句子里出现一个陌生符号名本身不算问题——只有当**不点开链接就无法理解本句在说什么**（符号的角色在句内完全悬空）时才记录。
- `<div class="callout">💡 进阶细节` 区块是刻意的进阶分区，面向回头查细节的读者，允许比正文密。**你的总评分只基于正文**（callout 之前的部分）；`unclear_sentences` 里**不要放 callout 区的句子**（它是可跳过的深细节，密是正常的）——只挑正文里读不懂的句子。给 callout 一个分节评分即可。

按以下步骤评估：

1. **复述测试**：通读笔记后，用你自己的话复述——这个文件为什么存在？它体现了哪两三个关键决定？复述不出来就如实写"无法复述"并说明卡在哪。复述只许基于笔记内容，禁止编造。
2. **读不懂句子清单**：逐句挑出你读不懂的句子。"读不懂" = 需要停下来重读两遍以上，或读完仍不确定它在说什么。每句给出原因分类：
   - `term_undefined` — 术语此前没立起、当句没解释、glossary 也查不到；
   - `too_dense` — 一句塞了多个各自独立的事实/概念；
   - `ambiguous` — 指代不明（"它/这/该"指什么不确定）；
   - `assumes_context` — 假设我知道某个我不知道的背景。
3. **先用后立术语清单**：笔记使用时既没有当场解释、之前也没立起、glossary 也没有的词。
4. **分节理解评分**（按笔记的标题分节，1–5）：5 = 一遍读懂；4 = 基本顺畅偶有停顿；3 = 反复读最终能懂；2 = 反复读仍半懂；1 = 完全不知所云。
5. **总评分**（1–5）= 你对"我现在敢说我理解了这个文件的角色和关键决定"的信心。
6. **具体度评分**（1–5）：这篇是靠**一个具体例子/一次真实调用走查**让你懂的，还是只在讲抽象机制和比喻？
   - **讲流程/讲多个概念的枢纽页**（如 HTTP 门面、能力注册表这类目录页）：5 = 有一次**从触发到结果的端到端真实走查**（如「浏览器点『提交订单』→ `POST /api/orders` → 网关验 session → 路由查表命中 `orders.create` handler → handler 调 service 落库」）；4 = 有走查但偏薄/不完整；**3 = 撒了真实路径/符号名，但没有一次端到端走查把它们串起来**（光有锚点不算走查）；1 = 全是比喻和机制名词（「柜台」「后门」「负责调度」），一个能指着看的真实链路都没有。**关键：真实路径名 ≠ 走查；一个流程枢纽页没有端到端走查，最高只能给 3。**
   - **类型/目录清单页**：点名真实类型/子项即算具体（不需要走查），据此正常给分。
   - **同时列出"该给具体例子却只给了抽象/比喻/半截走查"的地方**（`abstract_spots`）。小文件笔记（纯 re-export、单函数）不看这条。

**上层目录框架（drawer，可查资料）。** 下方"上层目录框架"是这篇笔记在 viewer 里的父目录链——读者点面包屑 / drawer 就能看到，视为随时可查。所以**文件页省略了这条链里已讲的共享概念，不算缺陷**（那是分工：目录页立框架、文件页填特有）——只要本文件**特有**的内容讲全、且这些共享概念在文中有点名（可点链接）即可，别因为"它没解释某个跨切术语是什么"就扣分（除非本页自己就是该术语的主页）。

**概念页专项（`__dir__` / 枢纽页尤其要查）。** 若本页引入一个命名抽象（如"capability""transform""frame"——本页像是这个概念的主页）：

- **定义在先**：在拿这个词讲机制之前，本页有没有用一句大白话 + 一个真实例子说清它**是什么**？**假装 glossary 不存在**——你能只凭本页一句话把它定义出来吗？不能，就在 `undefined_terms` 标出这个核心词、`overall` 扣分、`overall_rationale` 写明"核心概念用了却没在页内立起"（glossary 查得到不算数，主页就该自己立）。
- **别埋 lede**：本页最能让你"一下子懂了"的那句心智模型，出现在开头、还是埋在中段/结尾/根本没有？埋了或没有 → 扣分并指出它应该前置。

另外，遇到**"是 X 不是 Y"这类对比树靶子**（"是『没配=没门』，不是『有门但拒绝』""不是 A 而是 B""而不是…"——先断言再否定一个假想对立面，而那个对立面不带新信息），算**行文垃圾**：该节 `section_scores` 扣分，并在 `comment` 里引用原句点名"对比树靶子，是什么直接说"。例外：被否定项是读者会真假设、澄清它带来真信息的（如"fail-loud 而不是带病跑"），不算垃圾。

诚实优先：这是质量验收，不是礼貌场合，宁可苛刻不可放水。不要因为笔记"看起来专业、术语多"就给高分——唯一依据是你的真实理解程度。同样也不要为了显得严格而虚报问题：清单里每一条都必须引用笔记原句。

最后，只输出一个 JSON 对象，不要输出任何其它文字、不要用代码围栏包裹。**字段名必须与下面的骨架完全一致**（骨架里的中文是待填内容的说明）：

```
{
  "retell": "复述内容；无法复述时写明卡在哪",
  "retell_ok": true,
  "key_decisions": ["关键决定一", "关键决定二"],
  "unclear_sentences": [
    {"sentence": "笔记原句", "reason": "term_undefined|too_dense|ambiguous|assumes_context", "note": "一句说明"}
  ],
  "undefined_terms": ["先用后立的术语"],
  "section_scores": [
    {"section": "章节标题", "score": 4, "comment": "一句评语"}
  ],
  "overall_score": 3,
  "overall_rationale": "总评分的一句依据",
  "concreteness": 3,
  "abstract_spots": ["该给具体例子却只给了抽象/比喻的地方（原句或小节名）"]
}
```
