# 修订 prompt（拼接顺序：本文件 → 目标路径与笔记文件路径 → 盲读共识问题 → 事实核查问题 → 术语表）

你在修订一篇 codebase atlas 笔记。这篇笔记刚经过质量验收：3 位没有代码访问权的"盲读者"（扮演第一天入职的工程师）报告了读不懂的句子，1 位有代码访问权的事实核查员报告了与源码不符的断言。你的任务是**逐条修复这些问题**，产出一版能让陌生工程师一遍读懂、且每条断言都被源码支撑的笔记。

## 硬性边界

- **只允许修改一个文件**：下面给出的笔记文件（`.atlas/notes/...`）。仓库里任何其它文件——包括被描述的源码、别的笔记、glossary——一律不许动。这是多 agent 共享的工作区，越界写入会毁掉别人在飞的工作。
- 不许改笔记 frontmatter（`---` 之间的 hash/stamped 等），只改正文。
- 修订前先读目标源码文件本身，事实修正必须以当前源码为准，不许照抄源码注释里宣称的意图——以"代码实际做什么、谁真的调用它"为准。

## 修订原则

- **逐条修，不整篇重写**：没被投诉的段落保持原样。
- **低分章节允许结构性处理**：若某节被盲读者整体判为不合格（常见于只写「见上节 / 见源码」、没有独立信息量的薄指针节），你有三个选择——给它写出读者不点链接也能懂的独立内容；把它合并进承载同一内容的相邻节（然后删掉这个空壳）；或整节删除（如果它纯属和别处重复）。选哪个取决于这节是否真有值得独立成节的内容。合并/删除属于针对性修复，不算「整篇重写」。逐段导读的 `code:` 锚点若随薄节一起删除，确认被嵌入的类型在别处仍有链接可达。
- 修可读性问题时遵守这几条铁律：
  1. 术语先立后用——每个机制名词出场前，要么已在本页立起，要么当句用一句平实话解释，要么在 glossary 里（glossary 里有的词可以直接用）；
  2. 一句只装一个事实——被标 `too_dense` 的句子拆成多句或列表，拆完每句仍要完整通顺；
  3. 反事实句（"没有 X 就会 Y"）里的 X、Y 都必须是已立起的概念，否则先立概念再讲后果；
  4. 修辞降温——不加"爆炸半径"这类自造比喻；已有的自造比喻若被投诉，换成平实说法。
  5. **删掉"是 X 不是 Y"这类对比树靶子**——"是『没配=没门』，不是『有门但拒绝』""不是 A 而是 B""而不是"这种句式，是什么就直接说什么，砍掉那个被否定的假想对立面。**例外**：被否定项是读者会真的假设、且澄清它带来真信息的行为（如"启动期 fail-loud 而不是带病跑"点出了另一种真实行为），这种可留。默认删。
- 修事实问题时：`unsupported` 的断言按核查证据改写为源码支撑的说法；确实无法核实又不重要的句子直接删掉。若核查证据与你自己读码的结论矛盾，以你读到的当前源码为准，并在最终输出里说明。
- **事实密集漂移（≥3 条 unsupported）= 这篇笔记已和代码脱节，别再逐句打补丁**：打开被描述的源码文件通读一遍，把出问题的那几节按代码现实重写，而不是只替换被点名的句子——只补被点名的句子往往会在重写里带出新的不实断言，下一轮又被抓到（打地鼠）。
- **在上层目录框架下修（分工）**：下方给了从父目录到根的整条 `__dir__` 链。若本笔记重讲了目录页已立的共享概念，**删掉**（读者在 viewer 的 drawer 里能看框架）——只留本路径特有的；别把本文件特有内容也删了，也别用"见目录页"指针替代省略。
- **收尾自查（破打地鼠的关键，必做）**：改完后，把你写下的每一条事实断言（某函数做什么、谁调用它、字段/枚举语义、抛什么错、可选项后果……）逐条对照当前源码再核一遍——你有代码读权限，用它。任何一条你无法在源码里指到具体符号佐证的，要么改到能佐证，要么删掉。宁可少写一句，不可留一句没核实的断言。
- 反引号里的代码符号在阅读页面上会变成可点链接，`[标题](code:起始..结束)` 是内容锚点——保留原有锚点结构，除非那一段被删除。

## 完成后

输出一段简短总结：修了哪几类问题、有没有你不同意的投诉（说明理由）、有没有需要人工决断的遗留。不要输出笔记全文。
