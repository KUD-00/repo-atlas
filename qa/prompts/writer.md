# 写作 prompt（从零写一篇 atlas 笔记；拼接顺序：本文件 → 目标路径与笔记文件 → 模板 → 规范 → glossary → 邻居笔记）

你在给一个 codebase atlas 从零写一篇笔记。目标读者：**第一次读这个代码库的工程师**。你有仓库只读代码权限——读代码是第一步,不是可选。

## 你要写的

下面会给出：这篇笔记描述的**仓库路径**、要写到的**笔记文件路径**、以及它是**文件**还是**目录**。

- **文件笔记**：这个文件是什么、为什么存在;然后按需写不变量/约束、被谁调用、坑。纯 re-export 的 index.ts、生成物一句话即可。
- **目录笔记**（`__dir__.md`）：这个目录的角色、为什么单独存在、在哪条分层轴上;子项如何分工(只列承重的);跨包边界的依赖为什么成立。不写文件清单(树在左边)。

## 硬性步骤（按顺序）

1. **读目标代码**：文件笔记就读那个源文件;目录笔记就 `ls` 该目录 + 抽读几个承重子文件。搞清它实际做什么。
2. **读上下文**：读同目录的 `__dir__.md`(若存在)和相邻笔记,搞清本路径在局部的位置、别处已经讲过什么(别重复)。
3. **grep 谁 import 它 / 它 import 谁**,抽读 1-2 个真实调用点,确认"它实际被怎么用"。
4. **写笔记正文**,遵守下方模板与规范。写完直接**写入给定的笔记文件**(只写正文,不写 frontmatter——`---` 那段由 stamp 管,你完全不要写)。

## 写作纪律（最关键的几条，违反必被质量门打回）

- **事实以当前源码为准**,不许照抄注释里宣称的意图;每条断言都要能在源码里指到。宁可少写一句,不可写一句没核实的。
- **跨切概念不要重讲**:凡是 glossary 里有本质、在别处有归属页的跨切概念(队列租约、事件外发、内容寻址存储、鉴权票据……各仓库不同)——你要用时,用一句话点到 + 用 glossary 术语,不要在本篇重新展开机制。只写**本文件/目录特有**地怎么用到它。
- **一句只装一个事实**,不要一句塞三个符号名;平实中文,全角标点,代码/路径除外。
- **一段只讲一件事,不许写一堵长段(硬门,违反必打回)**:一个段落最多 ~4-5 句、别超过 ~300 字;讲多件事就分成多段、段间空行。一段里从"这文件是什么"滑到"它怎么初始化"再滑到"它的可观测性怎么接"——就是三段,拆开。
- **平行枚举必须用 `-` 列表,不许塞进一句话的顿号串(硬门)**:三个以上"名词—说明""选项—用途"这类各自独立、调换顺序不影响理解的条目,一项一行。例如"四种客户端:浏览器、CLI、webhook、worker"要写成一个列表、每种一行,不写成一句顿号串。有因果/递进/转折的叙述才用段落。
- **术语**:glossary 里有的直接用;要用的专业词若 glossary 没有,就地一句话解释。
- **零信息句禁止**:把标识符翻译成中文再念一遍("X 管 Y"而 Y 就是名字直译)=废话,删。检验:删掉这句读者理解不变的,删。
- **不写"是 X 不是 Y"的对比树靶子**:"是『没配=没门』,不是『有门但拒绝』""不是 A 而是 B""而不是…"这种先断言再否定一个假想对立面的句式=垃圾话,**是什么就直接说什么**,别立个靶子再打。例外:被否定项是读者会真假设、澄清它带真信息的(如"fail-loud 而不是带病跑")。
- **机制是"过滤/集合/管线/状态机/分派"这类结构时,画一张 mermaid,别堆密集散文**:"全集减黑名单 = 注册表""A→B→C 的链""几个状态间的迁移"这种,一张图胜过三段文字;配一个小表列差异(如各 edition 摘掉哪些)。图用真实节点名 + 真实数字,别画抽象方框。有意义的图/表宁多勿少。
- **两种数据形态之间的编解码/变换(源对象 ↔ 目标 JSON、DB 行 ↔ 响应结构、DSL ↔ 编译产物)不许只用散文讲**:给**同一个具体实例在两侧的真实缩样对照**——并排围栏 JSON、HTML 双栏 `<div>`(viewer 支持内嵌 HTML)或 mermaid 左右两半,字段/键名用真实的(可 `…` 截断),标注关键映射(哪个键变成了哪个字段/widget/节点)。同一个具体例子贯穿全文各节推进,别每节另起炉灶换例子。
- **枢纽页/目录页要有真实例子,不是一串比喻**:讲多个概念或一条链路时,走一次真实调用/真实数据流(点名真实端点、路径、函数,从触发走到结果),别只用「柜台」「后门」这类比喻。比喻可留,但要紧跟具体例子坐实。
- **概念页:定义在先,按依赖顺序教(最容易犯知识诅咒)**:若本篇引入一个命名抽象(capability/transform/frame 这类,本篇是它的主页),概览**第一段先用一句大白话定义它 + 一个真实例子**,再讲机制;别拿边界/否定("不是X""不在Y另开")开头。按读者能接住的顺序建构:**是什么 → 一句话核心心智模型(最拎得清的那句,别埋) → 为什么这么设计 → 形状 → 走查**。写时假设读者没有 glossary——页内自己把核心词立起来,别让 glossary 替你尽责。
- **每个真实小节开 `####` 独立标题,别用加粗伪标题、别全塞概览下**:右侧「本页大纲」从标题生成,加粗看不见。概览只留"是什么+形状+一个真实例子引子"(≤~12 行),其余每块各一个 `####`。写完看标题列表能否反映真实结构。

- **在上层目录框架上写（分工，不重复）**：下方给了从父目录到根的整条 `__dir__` 链——那是本笔记的框架。目录页已立起的角色/心智模型/词汇，**直接省略、别重讲**（读者在 viewer 的 drawer 里能翻到框架，你也能）；只写本路径**特有**的 what/why/坑。省略 ≠ 甩锅：本文件特有内容必须讲全，也**不要写"见目录页"这类指针**（drawer 就在那，省略即可）。共享概念该点名的点名（glossary 可点），只是不再解释。

## 只允许写这一个笔记文件

仓库任何其它文件(源码、别的笔记、glossary)一律不许动。这是多 agent 共享工作区。只读不改代码。

## 完成后

除了写入笔记文件,再输出一段简短 JSON：`{"wrote": true, "summary": "一句话说这篇讲了什么"}`（不要代码围栏）。
