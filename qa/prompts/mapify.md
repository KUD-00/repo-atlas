# 地图化 prompt（把"墙"改成"地图"；拼接顺序：本文件 → 目标路径与笔记文件 → 当前笔记正文 → glossary）

你在重构一篇 codebase atlas 笔记。它现在是**一堵墙**：内容准确，但**全部机制内联在同一个海拔**，单节几十行、整页密到没有循序渐进。你的任务：把它改成**地图**——读者先在 30 秒内看懂形状，想深入再往下走。你有仓库只读代码权限。

## 什么是"地图不是 territory"

- **概览**：这个文件/目录是什么、为什么存在、它的形状（几个阶段 / 几块，一句话列出；线性流程可配一张 `flowchart LR`）。读完概览就懂全局。
- **每个 section 只讲决策海拔**：这块**做什么、为什么这么摆**（2-5 句）。**机制细节不在这里展开**——指向承载它的地方：
  - 有子笔记的 → 一句话点到 + `见 <子笔记路径>`（如 `见 src/scheduler`）；
  - 没有子笔记家的深机制 → 下沉进文末「进阶细节」callout。
- **进阶细节 callout**（`<div class="callout"> 💡 **进阶细节** …`）：放读者初读可跳过、回头排查才看的深机制（字段级细节、边角判定、并发防重、循环依赖接法等）。callout 允许比正文密。

典型形状（以组合根/入口文件的笔记为例）：概览+流程图 → 每阶段一小段决策海拔 → 深机制全进 callout。

## 绝对禁令（最容易犯，违反直接返工）

- **不许把"这篇怎么读"写进笔记。** 笔记只讲被描述的代码，不讲自己的结构。禁止出现：「这一篇是地图，不是 territory」「按…讲决策海拔」「初读可跳过」「回头排查时查这里」「细节见文末」「文末进阶细节」这类**元话术 / 受众标签 / viewer 说明**。这些是给你（写作者）的指令，不是给读者的正文。callout 就在下面，读者自己看得见，不用正文去指「见文末」。
- **平行枚举必须用 `-` 列表，不许塞进一句话的顿号串。** 三个以上"名词—说明"、"选项—用途"这类各自独立、调换顺序不影响理解的条目，一项一行。例如"敲门的客人有四种：A、B、C、D"要写成一个列表，每种客人一行。读者要逐项扫，扫不了长句里的顿号串。（有因果/递进/转折的叙述才用段落。）

## 结构要能被大纲看见（每节独立标题，概览要短）

右侧「本页大纲」是从**标题**（`##` / `###` / `####`）生成的——所以：

- **每个真实小节开一个 `####` 独立标题，不要用加粗伪标题**（`**四种客户端**` 这种加粗行大纲看不见）。走查、每条链路、每个子系统、每类配置，各自一个 `####`。
- **「概览」只留：这是什么 + 形状（几块/几步）+ 一个真实例子引子，≤ ~12 行。** 别把整页塞进概览下。
- **概念页要"定义在先、按依赖顺序教"（知识诅咒重灾区）**：若本页是某命名抽象（capability/transform/frame 这类）的主页，概览第一段先一句大白话定义它 + 一个真实例子，再讲机制；别拿边界/否定开头。顺序：是什么 → 一句话核心心智模型（最拎得清的那句，别埋在中段）→ 为什么这么设计 → 形状 → 走查。假设读者没有 glossary，页内自己立起核心词。
- 检验：写完看看标题列表——**如果只有 1-2 个标题而正文很长，就是把内容闷在概览里了，拆成 `####` 小节**。大纲要一眼反映真实结构。

## 具体落地（枢纽页尤其重要，最容易被压成抽象）

- **地图要有一个能指着看的真实例子，不是一串比喻。** 概览或第一节里，走一次**真实调用/真实数据流**把抽象落地：点名真实的端点 / 路径 / 函数，从触发走到结果。例如别写「这层 API 是产品功能的柜台」，要写「浏览器点『提交订单』→ `POST /api/orders` → 网关验 session 写身份头 → 路由查表命中 `orders.create` handler → handler 调 service 落库」（换成你仓库里真实存在的那条链）。
- **禁止只用比喻/机制名词**（「柜台」「后门」「负责调度」「敲门的客人」）而不给一个真实例子。比喻可以有，但必须紧跟一个具体例子把它坐实。
- 数据流图（`flowchart`）画**真实的那条链**（真实节点名），不画抽象方框。
- **机制本质是"过滤/集合/管线/状态机/分派"时，把那段密集散文换成一张 mermaid + 一个小表**（如"全集减黑名单=注册表"配各 edition 摘掉哪些的表）。有意义的图/表宁多勿少——散文讲不清的结构，图一眼就清。

## 铁律

- **绝不丢事实——只搬家，不删。** 墙里的每条真信息要么留在正文（提到决策海拔）、要么下沉进 callout、要么指向已覆盖它的子笔记。不许因为"精简"就丢掉一个真判定。
- **事实以当前源码为准**（你有读权限），不许照抄注释意图；搬家时顺手核实。
- **一句只装一个事实**，平实中文，全角标点（代码/路径除外）。
- **一段只讲一件事，绝不留长段（硬门，质量门会因此打回）**：一个段落最多 ~4-5 句、别超过 ~300 字；一段里从"是什么"滑到"怎么装配"再滑到"可观测性"就是三段，段间空行拆开。原墙里的长段落是本次要消灭的主要目标之一。
- **在上层目录框架上重排（分工）**：下方给了整条 `__dir__` 链。目录页已讲的共享概念**省略、别重讲**（读者有 drawer 能看框架），只留本路径特有的；别写"见目录页"指针。这能让文件页普遍瘦一圈——正是地图化想要的。
- **删掉"是 X 不是 Y"的对比树靶子**："是『没配=没门』，不是『有门但拒绝』""不是 A 而是 B""而不是…"这种先断言再否定假想对立面的句式=垃圾话，是什么直接说什么。例外：被否定项带真信息（如"fail-loud 而不是带病跑"）。
- **跨切概念用 glossary 术语点到，不重讲**（凡在 glossary 有本质、别处有归属页的概念——队列租约、事件外发、内容寻址存储之类）。
- **保留 `code:` 锚点**：section 标题上的 `[名字](code:起始..结束)` 原样保留（除非整节删除）；概览的 mermaid 图保留或改进。
- **body 每个 section 目标 ≤ ~15 行**。超了就是还没搬够——继续把机制下沉。

## 笔记类型微调

- **文件笔记**：概览 + 按代码阶段/区域的决策海拔小段 + callout。
- **目录笔记（`__dir__`）**：概览（角色/为何单独存在/在哪条分层轴）+「分工」每子项一句 +「关系与不变量」+ callout 放跨子项的深机制。别写文件清单。
- **类型笔记（types.ts 等导出很多 interface）**：概览 + 「interface 总览」按逻辑簇分组（每簇一句业务决定 + `[簇名](code:起始..结束)` 锚点），**每个类型的字段级细节下沉 callout 或干脆不写**（字段自己会在预览里显示）。别一个类型一大段。

## 边界

- **只允许改这一个笔记文件**。源码、别的笔记、glossary 一律不动。只读不改代码。不改 frontmatter。

## 完成后

除了改写笔记文件，输出一段简短 JSON（不要围栏）：`{"mapified": true, "moved_to_callout": "搬进 callout 的大致内容", "summary": "一句话"}`。
